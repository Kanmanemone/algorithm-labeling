# 병합 판단 노트 (T006)

네 출처(solved.ac, LeetCode, Codeforces, 프로그래머스)의 범위 내(IN-SCOPE) 태그 167개(84+47+26+10)를 한 번에 비교해 61개 라벨(18개 카테고리)로 병합했다. (T009 자체 점검 중 solved.ac의 `lazyprop`·`flt`·`hackenbush`·`inclusion_and_exclusion`·`lte`, LeetCode의 `Bitmask`가 초안 분류에서 누락된 것을 발견해 보완함 — 상세는 sources/solved-ac-tags.md·leetcode-tags.md 참고.) FR-003 기본 규칙(애매하면 보수적 분리)을 적용했고, 각 라벨은 FR-002에 따라 정확히 하나의 카테고리에 배정했다.

## 병합 원칙

1. **동일 개념 → 병합**: 이름은 달라도 가리키는 알고리즘/자료구조가 같으면 하나의 라벨로 합친다. 예: solved.ac `dijkstra`/`shortest_path`/`floyd_warshall`/`bellman_ford` + CF `shortest paths` + LeetCode `Shortest Path` → **최단 경로** 하나.
2. **포괄 태그 → "일반" 라벨**: solved.ac `data_structures`/`graphs`/`graph_traversal`, CF `data structures`, LeetCode `Graph Theory`, 프로그래머스 `그래프`처럼 특정 기법을 가리키지 않는 포괄 태그는 억지로 세부 라벨에 끼워 맞추지 않고, 해당 카테고리의 "일반" 라벨(예: **자료구조 일반**, **그래프 이론 일반**)로 모은다.
3. **범위/난이도 차이 → 분리**: 예: 세그먼트 트리와 인덱스 트리(BIT)는 개념이 겹치지만(둘 다 구간 쿼리 자료구조) 구현 방식과 트레이드오프가 달라 별개 라벨로 유지했다.
4. **애매함 → 보수적 분리**: 예: `hashing`(해싱 기법)과 `hash_set`(해시 기반 집합/맵 자료구조)은 겹치는 문제가 많아 병합할지 고민했으나, 전자는 기법·후자는 자료구조 활용이라는 관점 차이가 있어 명확히 구분하기 어려운 경우로 보고 별개 라벨(**해싱**, **집합과 맵**)로 남겼다.
5. **새 카테고리 신설**: 원본 네 출처 어디에도 명시적 카테고리 구분이 없었지만, 61개 라벨을 정리하다 보니 자연스럽게 18개 상위 카테고리가 필요했다(예: "완전탐색과 백트래킹"을 그래프 카테고리와 분리해 별도 카테고리로 신설 — 완전탐색/백트래킹은 그래프에 국한되지 않는 범용 탐색 패러다임이라 판단).

## US2용 — "애매해서 보수적으로 분리" 또는 "범위/난이도 차이로 분리"로 표시된 유사 라벨 쌍 (T010 입력)

- **BFS vs DFS** (범위 차이): 둘 다 그래프 탐색이지만 방문 순서와 용도가 다름 → 판별기준에서 구체화 필요
- **스택/큐/덱 vs 모노토닉 스택/큐** (범위 차이): 후자는 단조성을 유지하는 특수 활용
- **세그먼트 트리 vs 인덱스 트리(BIT)** (범위 차이): 둘 다 구간 쿼리 자료구조지만 구현 복잡도·지원 연산 범위가 다름
- **해싱 vs 집합과 맵** (애매함, 보수적 분리): 관점 차이(기법 vs 자료구조 활용)로 겹치는 문제가 많음
- **이분 탐색 vs 삼분 탐색** (범위 차이): 전자는 정렬된/단조 조건, 후자는 볼록/오목(단봉) 함수에 적용
- **투 포인터 vs 슬라이딩 윈도우** (범위 차이): 후자는 전자의 특수 형태(연속 구간 유지)
- **완전탐색 vs 백트래킹** (범위 차이): 후자는 가지치기로 탐색 공간을 줄임
- **자료구조 일반 / 그래프 이론 일반 vs 각 세부 라벨** (포괄 태그 특성): "일반" 라벨은 특정 기법이 아니라 포괄적 활용을 가리킴을 판별기준에 명시 필요
