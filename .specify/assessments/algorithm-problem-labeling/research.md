# Idea Research: 알고리즘 문제 라벨링(유형화) 시스템

## Users & Demand

- Korean job-seekers preparing for technical/coding tests actively write and consume "문제 유형 정리" (problem-type roundup) guides and study-plan advice, e.g. by-type study blogs and job-prep community posts. This indicates real, ongoing demand for type-organized problem sets among 취업준비생. — [source: velog "코딩테스트 문제 유형 정리"](https://velog.io/@pppp0722/코딩테스트-문제-유형-정리); [링커리어 커뮤니티 "2025 코딩테스트 준비법"](https://community.linkareer.com/employment_data/4218903) (confidence: medium, cited)
- The most frequently cited "must-know" problem types for Korean coding tests are 구현(implementation), 완전탐색(brute force), and DP — a recurring, converging signal across multiple independent sources. — [source: search synthesis of velog/스터디 guides] (confidence: medium, cited)
- Demand is strong enough that it already supports paid commercial products (see Prior Art / Market), not just free community content — a stronger demand signal than requests alone. (confidence: high, cited)

## Prior Art

- **solved.ac** (community project layered on Baekjoon Online Judge/BOJ): maintains a hierarchical tag taxonomy (algorithm/data-structure/technique tags, where child tags imply parent tags) plus a per-problem difficulty tier (Bronze–Ruby) and per-tag "tag rating." This is the closest existing thing to "공신력 있는 유형화" for BOJ problems in the Korean ecosystem, though it is community-crowdsourced rather than an official BOJ product. — [source: solved.ac 태그/난이도 help pages](https://help.solved.ac/ko/problem/level) (confidence: high, cited)
- **solved.ac API**: an *unofficial*, community-documented API exists (CC0-licensed docs) exposing tags/tiers. The docs repo explicitly warns it "may be deleted at any time" by the original author and "may not match the current API," and notes a 2023 CORS restriction requiring a proxy/tool to call it from a browser. Not a stable foundation to build on as-is. — [source: solvedac/unofficial-documentation](https://github.com/solvedac/unofficial-documentation) (confidence: high, cited)
- **프로그래머스 (Programmers)**: officially curates a "코딩테스트 고득점 Kit" — a hand-picked, type-labeled problem set (스택/큐, 힙, 정렬, 완전탐색, 그리디, DFS/BFS, 이분탐색) explicitly built from analysis of what appears often / what people get wrong on real coding tests. This is an official, authoritative-source example of the exact labeling behavior the idea proposes, already built and free to use. — [source: 프로그래머스 코딩테스트 고득점 Kit](https://school.programmers.co.kr/learn/challenges?tab=algorithm_practice_kit) (confidence: high, cited)
- **LeetCode**: officially tags problems by topic (Array, DP, Graph, Tree, Two Pointers, Backtracking, etc.) and difficulty, and by asking-company (via third-party trackers). On top of that, multiple community projects re-curate LeetCode problems into hand-picked topic sets for interview prep, e.g. `berknology/leetcode-classifier` (14 categories, 259 problems, manually curated from two other existing curations, each entry annotated with difficulty + "main idea"). — [source: berknology/leetcode-classifier](https://github.com/berknology/leetcode-classifier) (confidence: high, cited)
- **Codeforces**: officially exposes both problem tags and a numeric difficulty rating via its public API (`rating` field on the Problem object), plus community tooling built on top of it. — [source: Codeforces blog on difficulty/tags](https://codeforces.com/blog/entry/62865); [codeforces-api client](https://github.com/yjiao/codeforces-api) (confidence: high, cited)
- **`tony9402/baekjoon`** (GitHub, 4,800+ commits): a large, actively maintained, manually-curated collection organizing 1,000+ BOJ problems into 23 algorithm categories, using solved.ac difficulty tiers and marking "worth studying" problems, with solutions in C++/Python/Java. This is close to a direct, already-existing prior-art implementation of the requested idea, at least for the BOJ corner of it. — [source: tony9402/baekjoon](https://github.com/tony9402/baekjoon) (confidence: high, cited)
- **Academic ML work on automatic tag/difficulty prediction**: multiple 2023–2024 papers attempt to *automate* tag and difficulty labeling from problem text/code (e.g., a 2,400-problem Codeforces dataset reaching ~78% tag-prediction accuracy with an MLP; an "AMT" dataset for joint tag+difficulty prediction; an 11,913-sample ALG-109 source-code classification dataset). This confirms the labeling task is nontrivial (multi-label, subjective) and that even specialized ML models don't yet reach reliably "exhaustive/authoritative" accuracy — full automation is a research problem, not a solved one. — [source: arXiv:2308.01863](https://arxiv.org/abs/2308.01863); [arXiv:2310.05791](https://arxiv.org/abs/2310.05791) (confidence: high, cited)
- **Codetree, AlgoExpert** (commercial): paid platforms already sell a "systematic problem-type classification + curriculum" product for coding-test/interview prep, built by algorithm specialists, explicitly targeting the same 취업준비생 audience. This is the most direct commercial prior art / competitor to the idea as stated. — [source: Codetree 학습 안내](https://www.codetree.ai/ko/trail-info); [Codetree Medium post](https://medium.com/@codetree_team/알고리즘-국가대표가-만드는-전-세계에-단-하나뿐인-코딩-교육-서비스는-6f6c8e60cf0d) (confidence: high, cited)

## Market & Context

- Today, a job-seeker who wants "all problem types, sourced from authoritative sites, comprehensively labeled" already has to stitch together multiple *free, official, per-platform* taxonomies (solved.ac tags, Programmers Kit, LeetCode topics, Codeforces tags) that don't share a common schema — or pay for a commercial product (Codetree, AlgoExpert) that already did the unification work for them. The "cost of doing nothing" is mainly the manual effort of cross-referencing sources, not a total absence of labeled material. (confidence: medium, ASSUMPTION synthesized from the Prior Art findings above — no single source states this directly)
- The 2026 "코딩테스트 사이트 추천 BEST 5" roundup blog treats problem-source choice (백준/프로그래머스/LeetCode/etc.) as already a mature, well-covered topic for this audience, reinforcing that this is a well-trodden space rather than a greenfield gap. — [source: 코드트리 블로그 "2026 코딩테스트 사이트 추천 BEST 5"](https://www.codetree.ai/blog/ko/2026-코딩테스트-사이트-추천-best-5) (confidence: medium, cited)

## Data & Constraints

- **BOJ (acmicpc.net) usage rules** explicitly prohibit using LLMs or automated/external-program submission tools for solving/submitting problems (violation results in submission deletion and possible access restriction). This rule is about automated *submission*, not read-only scraping of problem text/tags, but it signals the operator actively polices automated tooling on the site. The rule page itself (`help.acmicpc.net`) was not directly fetched — it is not on this workflow's pre-approved host allowlist — so this finding is based on search-engine-surfaced excerpts, not the primary text. — [source: search excerpt of BOJ 이용 규칙](https://help.acmicpc.net/rule) (confidence: medium, cited via search snippet — **not independently fetched, verify directly**)
- **General Korean crawling case law**: legal analyses converge on "크롤링 자체가 불법은 아니지만, 이용약관상 명시적 금지·기술적 접근 제한을 위반하거나 수집물을 그대로 복제·재배포하면 저작권법/부정경쟁방지법 위험이 있다" (crawling itself isn't illegal, but violating an explicit ToS prohibition, bypassing technical access controls, or verbatim republishing collected content carries copyright/unfair-competition risk). Aggregating and *re-expressing* problems as structured labels (rather than republishing full problem text) is more defensible, but per-site ToS still needs direct review. — [source: 크몽 크롤링 법률 가이드](https://kmong.com/article/3132--크롤링-의뢰-법률-문제-없이-안전하게-진행하는-방법); [nepla.ai 크롤링 법적 쟁점](https://www.nepla.ai/nest/@xxWIvIKZwKx7/free/크롤링-스크래핑-에-관한-법적-쟁점과-크롤링시-고려-사항) (confidence: medium, cited — general legal commentary, not a ruling specific to BOJ/Programmers)
- Neither `acmicpc.net`'s nor `programmers.co.kr`'s actual Terms of Service text was fetched in this pass (both hosts are outside the workflow's pre-approved allowlist, requiring explicit confirmation before fetching). Their specific crawling/redistribution clauses remain unverified. (confidence: n/a — explicit gap, see below)
- The most durable, lowest-risk data path found is **using official/semi-official APIs where they exist** (Codeforces' official API with a `rating` field; solved.ac's community-documented — but explicitly unstable and unofficial — API) rather than scraping HTML directly. (confidence: medium, cited)

## Evidence Against the Idea

- **The space is already crowded with credible, free, and paid solutions**: solved.ac (community tags for BOJ), Programmers' own 고득점 Kit, LeetCode's own topic tags plus multiple community re-curations, and commercial products (Codetree, AlgoExpert) all already do some version of "collect problems from authoritative sources and label them by type" for this exact audience. A new effort needs a clear differentiator (e.g., cross-platform unification, openness, a novel taxonomy, automation) to avoid being a redundant clone. — [source: see Prior Art citations above] (confidence: high, cited)
- **Full automatic labeling is an open research problem, not a solved one**: published deep-learning approaches to tag prediction top out around ~78% accuracy on a curated 2,400-problem set; "exhaustively" and reliably auto-labeling "all problems job-seekers encounter" likely requires substantial human curation/QA on top of any automation, which is a nontrivial ongoing cost. — [source: arXiv:2308.01863] (confidence: high, cited)
- **Legal/ToS exposure is a real, currently-unverified risk**: if the plan involves bulk-scraping BOJ/Programmers/LeetCode problem text (as opposed to using official APIs or manually-referenced tags), Korean case law and each platform's ToS could expose the project to takedown requests or civil liability, especially at any commercial or public-redistribution scale. — [source: see Data & Constraints citations above] (confidence: medium, cited)
- **"Authoritative" labels are themselves contested**: even solved.ac's tags — arguably the most "공신력 있는" Korean source — are community-crowdsourced consensus, not an institutional/official BOJ standard, and different platforms (BOJ/Programmers/LeetCode/Codeforces) use incompatible taxonomies and difficulty scales, so "the" authoritative unified labeling this idea wants doesn't currently exist anywhere — it would have to be synthesized and would inherit editorial judgment calls. (confidence: medium, ASSUMPTION drawn from the taxonomy differences observed above)
